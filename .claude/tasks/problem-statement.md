# Problem Statement: Side Effect Analysis for Simplicity Programs

## Goal

Analyse a compiled Simplicity program to understand its **side effects** — specifically what parts of the transaction environment each branch reads, and whether each branch can fail. This enables two classes of static analysis:

1. **Completeness check**: every branch that can be taken should have at least one write side effect (a failure/assertion). Branches with no failure path are suspicious — they allow the branch to be satisfied trivially.
2. **Optimisation**: branches that are provably unreachable or redundant (e.g. duplicated read sets) can be pruned.

---

## Definitions

### Write Side Effects (Fail/Assert)

A **write side effect** is any point in the program that can cause the script to fail. These are generated by:

- An explicit `Fail` node in the Simplicity AST
- A jet that performs a **verify** operation (e.g. `CheckSigVerify`, `CheckLockTime`)

Each jet will expose a method indicating whether it produces a write side effect (i.e. can cause script failure). `Fail` nodes always produce a write side effect.

### Read Side Effects (Transaction Introspection)

A **read side effect** is any piece of transaction data that a jet reads from its `Environment`. Each jet will expose a method returning the set of transaction fields it reads.

Transaction fields are **hierarchical**: a high-level jet like `sig_all_hash` implicitly reads many sub-fields (version, locktime, all inputs, all outputs, etc.). For analysis purposes this hierarchy is **denormalized** — each jet returns the full flat set of primitive transaction fields it reads, not just the top-level name.

---

## Required Changes to the Jet Interface

Add two methods to the `Jet` trait (or as a supplementary trait):

```rust
/// The set of primitive transaction fields this jet reads from the environment.
/// Returns an empty set for jets that do not access the environment.
fn read_effects(&self) -> &[TransactionField];

/// True if this jet can cause script failure (write side effect).
fn has_write_effect(&self) -> bool;
```

`TransactionField` is a flat enum of all primitive transaction data points:

```rust
pub enum TransactionField {
    Version,
    Locktime,
    InputCount,
    OutputCount,
    InputPrevOutpoint(InputSelector),
    InputSequence(InputSelector),
    InputValue(InputSelector),
    InputScriptSigHash(InputSelector),
    InputAnnexHash(InputSelector),
    OutputValue(OutputSelector),
    OutputScriptHash(OutputSelector),
    // ... Elements extensions: asset, nonce, issuance, etc.
}

pub enum InputSelector { Current, Any }
pub enum OutputSelector { Any }
```

Jets like `sig_all_hash` return the full denormalized list (version, locktime, every input field, every output field). Jets like `CurrentIndex` return only `[TransactionField::InputCount]` (or similar). Jets that do no environment access return `[]`.

---

## Effect-Annotated AST

Build a parallel or annotated AST where each node carries:

```rust
pub struct EffectSummary {
    /// All transaction fields read by this subtree (union of all jets in subtree)
    pub reads: BTreeSet<TransactionField>,
    /// Whether this subtree can ever fail (has any write side effect)
    pub can_fail: bool,
    /// Whether this node's output type is Unit and its subtree has no side effects,
    /// making it a candidate for pruning (replacement with a bare Unit node)
    pub is_pure_unit: bool,
}
```

This is computed bottom-up over the existing `Inner` node tree:

- `Jet(j)` → `{ reads: j.read_effects(), can_fail: j.has_write_effect(), is_pure_unit: false }`
- `Fail(_)` → `{ reads: {}, can_fail: true, is_pure_unit: false }`
- `Comp(s, t)` → union of both children; `is_pure_unit` is true iff the node's return type is `Unit` and neither child has any effects
- `Case(l, r)` / `AssertL` / `AssertR` → annotate each branch independently, then union for the full subtree; each branch is independently flagged with `is_pure_unit`
- `Iden`, `Unit`, `Word`, `Witness` → `{ reads: {}, can_fail: false, is_pure_unit: return_type == Unit }`

---

## Analyses Enabled

### 1. Missing Write Effects Per Branch

Walk the annotated `Case` / `AssertL` / `AssertR` nodes. For each branch, check `can_fail`. A branch with `can_fail == false` is a candidate for a completeness warning — it can be taken without any script failure, meaning the script does not enforce any condition on that path.

### 2. Transaction Field Coverage (Malleability Analysis)

Collect the union of all `reads` across the entire program. Compare against the full set of `TransactionField` values. Any field **not** in the union is not committed to by the script — a third party could malleate that field without invalidating the script. This is the malleability surface.

For example, if no jet reads `InputAnnexHash(Any)`, the annex can be modified freely. If `sig_all_hash` is used, all fields are covered and malleability is minimal.

### 3. Branch Pruning / Optimisation

Two branches of a `Case` node with identical `reads` sets and the same `can_fail` outcome may be candidates for merging or pruning, since they are observationally equivalent from an effect standpoint.

#### Unit-Type Pruning

Any node whose **return type is `Unit`** and whose entire subtree has **no side effects** (`reads` is empty and `can_fail` is false) is observationally inert — it computes nothing observable. Such a subtree can be replaced with a bare `Unit` node without changing program semantics.

This is tracked via `is_pure_unit` in `EffectSummary`. The pruning rule is:

```
if node.return_type == Unit && !summary.can_fail && summary.reads.is_empty() {
    replace subtree with Unit
}
```

This interacts with branch analysis: if one arm of a `Case` is `is_pure_unit`, that branch contributes nothing to the program's constraints — it is a silent no-op path. Combined with the missing-write-effects check, a branch that is both `can_fail == false` and `is_pure_unit == true` is definitively dead from a policy standpoint and can be reported to the user as a prunable branch.

The `is_pure_unit` flag therefore serves two purposes:
1. **Direct optimisation**: safe to replace with `Unit` during compilation or program simplification.
2. **Diagnostic signal**: helps identify branches that have no observable purpose, surfacing policy gaps or authoring mistakes.
